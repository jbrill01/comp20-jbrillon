Abstract of Findings

Issues Found

1. Cross-Site Scripting

  • Issue: Cross-Site Scripting
    • Cross-Site Scripting (XSS) is when the server takes in javascript as data and subsequently executes it.
  • Location: POST /sendLocation
    • Specifically, the three parameters taken in on POST (login, lat and lng) are neither regulated nor escaped. Therefore, anything – such as code contained in script tags – can be read in.
  • Severity of Issue: High
    • XSS is often ranked among the top security threats to websites, and that's because the Javascript taken in can do virtually anything, from alert messages (which I chose to do), to redirecting the window to corrupt websites.
  • Proof of Vulnerability
    • This is the alert message I put up on Jay's page
  • Resolution
    • Possible resolutions including limiting the input to alphanumerical only, or more preferably changing questionable characters into HTML versions (i.e. angle-brackets can become &lt; and &gt;).

2. Request Injection

  • Issue: Request Injection Attack
    • A request injection attack is when the a request.
  • Location: GET /locations.json
    • Specifically, the query that is taken in on GET is capable of taking in [$gt] and [$ne]. The first will return all data after a given input, and the second will return all data that is not a given input.
  • Severity of Issue: High
    • In this particular scenario, all database information is revealed just by pairing [$gt] with a blank input, or by pairing [$ne] with an input not in the database. Therefore, so long as you know the name of the requested query parameter and regardless of whether you know anything about the database itself, you can get the returned information for all members of the database.

4. Non-numerical Inputs

  • Issue: Non-numerical Inputs
    • Latitude and longitude should be numerical inputs (i.e. 43.21 or -123.45). However, Jay's application can take any input (as seen with issue number one), not just numerical ones.
  • Location: POST /sendLocation
    • When taking in the three parameters on POST (login, lat and lng), lat and lng have no regulations about what they have to be. Therefore, they are not limited to numerical characters.
  • Severity of Issue: Low
    • Beyond potentially crowding the database with input much longer than a standard lat and lng, the biggest problem this causes is a list of locations that are not actually locations. For instance, some of the locations on Jay's app are now just random characters, or they are blank. Ming noted this when storing a lat of "this" and a lng of "isbad".
  • Proof of Vulnerability
    • Here are two example non-numerical inputs from Jay's page
  • Resolution
    • One possible resolution would be to create an additional if statement, and say that if (isNaN(the inputted lat) || isNaN(the inputted lng)), send an error and do not store the input in the database
